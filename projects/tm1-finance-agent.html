Project 01 / Agentic AI for finance

TM1 Finance Agent

Conversational finance assistant that lets executives ask natural-language questions — "What was Profit and Loss for Q1 2026?" — and get proof-first, chart-backed answers straight from the company's official planning hierarchy.

01 Problem

Leadership needs fast, trustworthy answers to finance questions without waiting on an analyst to hand-write a query against a complex planning hierarchy. Every ad-hoc question — "what was gross profit last quarter," "how does this compare to plan" — turned into a multi-hour turnaround, and the answer's trustworthiness depended entirely on whoever wrote the query.

02 Solution

A chat application backed by a skill-based orchestrator — Intent Classifier, then Planner, then Step Executor — that resolves natural language into the correct hierarchy nodes and time periods, queries Snowflake, and returns an evidence-first answer: proof of exactly what was queried, a narrative summary, bullet insights, and suggested follow-ups. The agent keeps conversational memory across turns, so a follow-up like "compare against prior year quarter" resolves correctly without the user restating context.

03 Architecture

A user question enters the Orchestrator, which runs it through three stages — Intent Classifier, Planner, Step Executor — before handing off to three supporting services: Taxonomy, Synonym, and Time Resolution. Those services translate business language into exact hierarchy nodes and periods, which the Step Executor uses to query Snowflake directly.

07 Why a knowledge graph

Finance data like this doesn't live in a tidy table — it's a 7-level parent-child hierarchy (CLASS6 → CLASS5 → CLASS4 → CLASS3 → CLASS2 → CLASS1 → ACCOUNT), with six more dimensions slicing across it. Point a generic text-to-SQL tool at that and it has no idea Operating Income contains Gross Profit, which contains Trade GP, which contains — the rollup relationships are invisible to it.

So the agent doesn't flatten the hierarchy into a table or a semantic-layer YAML — it loads the whole thing as a knowledge graph at startup: nodes are financial concepts, edges are "rolls-up-to" links, and every node knows its parent, children, and full path. When a question comes in, the agent resolves it straight to a node and reads that node's neighborhood — parents for the rollup, children to drill — without ever walking from the root. That's GraphRAG-style grounding: graph traversal, not vector similarity search.

The model never writes SQL. It picks a tool and arguments; a vetted template turns that into a deterministic, whitelisted query — same question, same query, every time — with a proof line (metric · level · period · source) showing exactly what ran.

Why not just point a generic semantic-layer tool at it instead? Those map questions against a flat model in a YAML — tables, columns, joins. A 7-level recursive parent-child hierarchy doesn't fit that shape: flatten it and the drill relationships are lost, and generic text-to-SQL is non-deterministic enough that it can be silently wrong. Grounding the agent in the real graph structure — and giving it a skills playbook instead of just a system prompt — is what makes the hierarchy answerable.
