Project 01 / Internal platform engineering tool

Triton

Full-stack platform that reconciles approved fuel supply contracts against real transaction data, closing the loop between deal terms and actual performance.

Working local prototype
FastAPI SQLAlchemy 2.0 Alembic Postgres React 19

01 Problem

Fuel supply contracts get approved in an external system that only captures high-level terms — price, volume commitment, counterparty. Once a deal is signed, there's no system that tracks whether the contract is actually performing: no way to see contracted-vs-actual volume, no margin visibility, no gross profit rollup against the terms that were negotiated. The deal desk and finance team had no shared source of truth once ink hit paper.

02 Solution

Triton is a FastAPI + React application that ingests approved contract headers, lets the deal team maintain detailed bid lines against each contract, and automatically maps every line to real transaction records. The result is a performance report that shows contracted terms next to what actually happened — volume, margin, and gross profit — with visibility scoped to each user's position in the org hierarchy and enforced on the server for every single query, not just in the UI.

03 Architecture

Snowflake acts as the read-only source of truth for actual transactions and also receives BI exports back out. Five scheduled sync jobs move data between Snowflake and a Postgres operational store (Snowflake Postgres in production), which backs a FastAPI API secured with JWT auth. A React single-page app consumes that API across four functional surfaces: Bids, Mapping, Dashboard, and Admin.

Fig. 1 — Data flows left to right: Snowflake actuals → scheduled sync jobs → Postgres → FastAPI (JWT + RLS) → React SPA.

04 Tech stack

FastAPI SQLAlchemy 2.0 Alembic sqladmin APScheduler Postgres React 19 Vite TanStack Query MUI / MUI X Data Grid Auth0 (planned) Snowflake

05 Highlights

  • Five scheduled data-sync jobs keeping the operational store aligned with Snowflake actuals.
  • Server-side org-subtree row-level security enforced on every query, not just the UI.
  • Admin panel built on sqladmin for internal data management without custom tooling.
  • Clean JSON + JWT API contract between the FastAPI backend and React frontend.

06 Status

Working local prototype — the core flow (bids, LIFT mapping, dashboard, org tree + row-level security, access model) is built and running end-to-end. Snowflake integration and Auth0 login are the remaining phases before a production rollout. A live demo isn't available since the app connects to internal company data in Snowflake.
